# v5.3 Optimization Audit — 2026-04-21

Post-delivery review of the v5.3-EXEC sprint. Lists every known rough edge after
the three successful PRs (#18, #19, #20) landed. Each item is triaged into one
of four severities and carries an explicit **owner**, **acceptance criteria**,
and **evidence** (file path or curl command) so it can be worked and closed
without re-discovery.

**Legend**:

- **P0** — security / compliance / data-integrity — land before Pilot 2026-05-15.
- **P1** — functional parity with spec — land in Pilot W1 (05-15 → 05-22).
- **P2** — quality-of-life + performance — land in Pilot W2 / post-Pilot.
- **P3** — nice-to-have — no deadline.

Items marked ✅ FIXED in-session are already closed by the same commit that

## P0 — Security / Compliance / Data Integrity

### ✅ P0-SEC-1 — `/api/attribution/redemption` was publicly writable

**Risk**: any Internet caller could POST to this route and insert arbitrary rows
into `push_transactions`. The route comment claimed the gate existed; it did
not, because `middleware.ts` matcher only covered `/api/internal/:path*`.

**Evidence**: pre-fix prod `curl -X POST https://push-six-flax.vercel.app/api/attribution/redemption` returned 500 (error on DB call, not 403).

**Fix (this session)**: middleware.ts now gates `/api/attribution/*` with the
same `x-internal-api-secret` shared-secret check. Re-tested: `curl -X POST`
without header → 403.

### ✅ P0-SEC-2 — 5 test rows in prod `push_transactions`

**Risk**: fake-UUID test rows mix with real Pilot data, skew any aggregate
query (count, conversion rate, fraud metrics) during the first week.

**Evidence**: `SELECT count(*) FROM push_transactions` returned 6 before
cleanup — all fake IDs like `00000000-0000-0000-0000-000000000001`.

**Fix (this session)**: DELETE WHERE transaction_id IN (...) executed via
Supabase Management API. Table returned to zero rows before Pilot launch.

### ✅ P0-SEC-3 — No rate limit on `/api/attribution/redemption`

**Risk**: if `INTERNAL_API_SECRET` ever leaks (Vercel env dump, misplaced
webhook config, leaked agent prompt), an attacker could write millions of rows
in minutes. A single merchant POS rarely exceeds 20 redemptions / minute.

**Fix (this session)**: added `rateLimit("attribution-redemption:${ip}", 60, 60_000)`
at the top of the POST handler, returning 429 when exceeded.

### ✅ P0-BUG-1 — `/api/merchant/redeem` hardcoded `creator_id = merchantId`

**Risk**: the merchant POS path wrote `creator_id = gate.merchantId` as a
placeholder, which corrupts the ground-truth label (the row represents
merchant→merchant attribution instead of creator→merchant). This breaks every
downstream earnings / rank / cultivation computation.

**Evidence**: `app/api/merchant/redeem/route.ts:178` TODO comment.

**Fix (this session)**: resolve via `campaign_applications` join —
`SELECT creator_id WHERE campaign_id = X AND status = 'accepted' ORDER BY created_at DESC LIMIT 1`.
Falls back to merchant_id only when no accepted creator exists (flagged row).

### P0-SEC-4 — `consent_events` IP / UA columns never populated

**Risk**: CCPA/GDPR audit requires proof of consent under the exact conditions
consent was given. We added `ip_address_hash` and `user_agent_hash` columns
but no writer populates them — the audit table is a compliance placeholder,
not a working ledger.

**Evidence**: `grep -rln 'consent_events' app/ lib/` → only the service-role
client constant, no `.insert` call.

**Owner**: next session.

**Acceptance**: `/scan/[qrId]` → on consent tier change, POST to a new
`/api/internal/consent-events` endpoint that hashes IP + UA and inserts.
Backfill not required (pre-Pilot launch).

### P0-SEC-5 — Age gate is single unverified checkbox

**Risk**: COPPA (13-year threshold) requires "actual knowledge" — a single
self-serve checkbox satisfies CCPA for 13–17 but is weaker than what COPPA
literally requires for ≤12. The consent_tier downgrade is correct but the
detection path is soft.

**Fix path**: disclaimer language update + (optional) parental email
verification for anyone who checks the box AND claims to be under 13. For the
2026-05-15 Pilot we stay with the soft path and document the known limit.

**Owner**: legal review + next session.

### P0-DATA-1 — `SUPABASE_SERVICE_ROLE_KEY` is the legacy JWT

**Risk**: Supabase is migrating to `sb_secret_*` opaque tokens. The legacy JWT
we installed has `exp: 2091590167` (2036) — fine for years, but we should
migrate on our terms before Supabase forces it.

**Evidence**: `vercel env pull` dump shows `SUPABASE_SERVICE_ROLE_KEY` starts
with `eyJ...` (JWT) while `SUPABASE_SECRET_KEY` starts with `sb_secret_...`.

**Fix path**: generate a new `sb_secret_*` via Supabase Dashboard, update
Vercel env, verify `@supabase/supabase-js@^2.103.0` recognizes it, then
revoke the legacy JWT.

**Owner**: next session.

**Rotation policy** (new): rotate every 90 days, or immediately if any env
dump leaves a trusted surface.

## P1 — Spec parity for Pilot W1

### P1-FUNC-1 — `/scan/[qrId]` still reads `mock-qr-codes` instead of `qr_codes` table

**Impact**: the real `qr_codes` table exists but the consumer landing page has
no reader. Merchants can create a campaign + QR in the dashboard → can't
actually be scanned by a consumer.

**Fix path**: replace `getQRCode(qrId)` with a `db.select("qr_codes", { id })`
call guarded by `.eq("disabled", false)`. Expected scope: <30 lines.

### P1-FUNC-2 — `/api/merchant/qr-codes/[id]/sticker` still reads mock

**Fix path**: same mapping, but server-side (route.ts).

### P1-FUNC-3 — Oracle audit is NOT auto-written during normal redemption

**Impact**: only the admin manual-trigger path writes `oracle_audit` rows —
the normal POS / attribution flow skips it. Weekly decision-audit reporting
(spec P1-4) will have a sparse table.

**Fix path**: in both `/api/attribution/redemption` and `/api/merchant/redeem`,
after a successful insert, call `oracleService.saveOracleAudit(txId, result, { triggeredBy: "auto" })`.

### P1-FUNC-4 — `/admin/oracle-trigger` page is unlinked from admin nav

**Impact**: admin has no breadcrumb / nav entry to reach it — they must type
the URL manually. OK for week-1 ops, ugly for week-2+ handoff.

**Fix path**: add a nav row in `app/(admin)/admin/page.tsx` header.

### P1-DATA-1 — `qr_codes` missing uniqueness on (campaign_id, poster_type)

**Impact**: merchants can accidentally duplicate a poster for the same
campaign, splitting scan counts across two rows.

**Fix path**: migration — `ALTER TABLE qr_codes ADD CONSTRAINT
qr_codes_unique_per_poster UNIQUE (campaign_id, poster_type);`.

### P1-DATA-2 — Privacy-requests admin UI missing

**Impact**: the table has a 45-day SLA (`due_at`) but no admin queue to
surface due-soon / overdue requests. Legal exposure.

**Fix path**: build `/admin/privacy-requests` listing rows ordered by
`due_at ASC, status != 'resolved'`. Add simple "mark resolved" button.

## P2 — Quality-of-life / Performance

### P2-PERF-1 — `qr_codes` prefix lookup uses `ilike`, no trigram index

Prefix search scales O(N) without a `gin_trgm_ops` index. Fine for 100 QRs,
painful at 10k+.

**Fix path**: migration — `CREATE EXTENSION IF NOT EXISTS pg_trgm; CREATE INDEX
qr_codes_id_prefix ON qr_codes USING gin (id gin_trgm_ops);`.

### P2-PERF-2 — Sticker PDF regenerated every request

A single QR code's PDF is immutable until hero_message / sub_message changes.
Could be cached at the edge or memoized in the Lambda.

**Fix path**: add `cache-control: public, max-age=3600` (short TTL); full
solution is Vercel edge cache by `(merchant_id, qr_id, size)` tuple.

### P2-UX-1 — Merchant POS redeem page has no "scan QR → auto-fill prefix"

Today the cashier has to type the prefix. Most POS terminals have a camera;
support `getUserMedia` + a lightweight QR decoder (jsQR).

### P2-TEST-1 — E2E runs only against `localhost:3000`

No scheduled CI job smoke-tests prod. Drift between local pass + prod 500
goes undetected until next manual smoke.

**Fix path**: GitHub Action running `npm run e2e` nightly against
`PLAYWRIGHT_BASE_URL=https://push-six-flax.vercel.app` with a bypass cookie
provisioned via Vercel Access Token.

## How to close an item

1. Create a PR titled with the item id: `fix(v5.3): P0-SEC-4 consent_events writer`.
2. Update this file — flip the heading prefix from `### P0-SEC-4` to `### ✅ P0-SEC-4`
   and append a `**Fix**: <commit>` line referencing the PR.
3. Merge. Do not move items between severity levels without a note.
